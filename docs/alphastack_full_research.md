## 2.6 Market Microstructure
**File:** `research_market_microstructure.md`

### Key Findings
- Order book imbalance (OBI) > +0.5 → strong buy pressure; < -0.5 → strong sell pressure
- Level 1: Best bid/ask (free); Level 2: Full order book ($5-50/mo); Level 3: Institutional only
- Forex is OTC/decentralized — no visible order book like equities
- **Spread cost at $7:** EUR/USD 1.4 pips = $0.14 = **2% of capital per trade**
- News slippage: NFP/CPI can gap 20-50 pips = $3 = **43% of capital** on $7 account

### Order Book Mechanics
- Bids (buy side) sorted descending; Asks (sell side) sorted ascending
- Mid-price = (Best Bid + Best Ask) / 2 — "fair value" estimate
- Iceberg orders mask true depth; only partial size visible
- **Key insight for retail:** Level 2 (DOM) on MT5 ECN accounts provides significant edge

### Market Impact
- Institutional orders move the market: $100M EUR/USD = 1-3 pips impact (normal), 5-15 pips (volatile)
- Information leakage: The act of seeking liquidity signals intent
- **Last look:** Many FX venues allow LPs 50-200ms to reject trades — costs institutional clients $2-5B annually

## 2.7 Market Regime Analysis
**File:** `research_market_regime.md`

### Key Findings
- Four canonical regimes: Bull/Trending Up, Bear/Trending Down, Sideways/Range-Bound, Crisis/High Volatility
- A regime-aware system routing capital to appropriate strategies returned **18% (Sharpe 1.6)** vs ~0% for fixed allocation
- **ADX > 25** = strong trend; **< 20** = ranging
- HMM with 3 states is the quant standard for regime detection

### Detection Methods
1. **Rules-Based** (recommended starting point): ADX, realized volatility, 20-day return, cross-asset correlation, VIX
2. **Hidden Markov Model (HMM):** Quant standard; estimates state probabilities and transition matrix
3. **ML-Based:** XGBoost/LightGBM on engineered features; highest accuracy but black-box

### Decision Tree
```
1. IF volatility > 35% AND correlation > 0.8 → CRISIS
2. ELSE IF ADX > 25 AND |return| > 5% → TRENDING
3. ELSE IF ADX < 20 AND volatility < 15% → RANGE-BOUND
4. ELSE → TRANSITION/UNCERTAIN (reduce exposure)
```

### Strategy Allocation by Regime
| Regime | Trend Following | Mean Reversion | Breakout | Momentum |
|--------|----------------|---------------|----------|----------|
| Bull Trend | 60% | 10% | 20% | 10% |
| Bear Trend | 10% | 20% | 30% | 40% |
| Range | 10% | 60% | 20% | 10% |
| Crisis | 0% | 20% | 10% | 70% (defensive) |

## 3.3 Loop & Multi-Agent Systems
**File:** `research_03_loop_multiagent_systems.md`

### Key Findings
- **ReAct Loop:** Thought → Action → Observation → Thought — forces auditable reasoning chains
- **Reflection Loop:** Generate → Critique → Revise — self-correction cycle
- **Deliberation Loop:** Multi-step reasoning weighing multiple options before committing
- **Multi-Agent Frameworks:** CrewAI, AutoGen, LangGraph, OpenClaw — production-grade orchestration

### Loop Patterns for Trading
1. **ReAct:** Agent reasons about market, takes action (queries API), observes result, reasons again
2. **Reflection:** Pre-trade self-critique catches blind spots; post-trade analysis improves strategy
3. **Deliberation:** Weighs multiple trade options with pros/cons before committing
4. **Planning:** Decomposes complex strategies into sub-tasks executed sequentially

### Multi-Agent Architecture
- **Role-Based:** Separate agents for fundamental, technical, sentiment, risk analysis
- **Hierarchical:** Senior agents (portfolio managers) delegate to junior agents (researchers)
- **Market Ecology:** Multiple independent agents competing — raises crowding concerns

### Recommended for Alpha Stack
- Start with **2-3 agents** (research, analysis, execution)
- Use proven frameworks (OpenClaw, CrewAI) for orchestration
- Human-in-the-loop for high-stakes decisions
- Agent monitoring agents for quality control

- **TimescaleDB:** Time-series extension for OHLCV data, tick storage
- **Redis:** In-memory cache for real-time prices, session state, pub/sub messaging
- **ClickHouse:** Column-oriented for analytical queries on historical data
- **Schema design:** Normalized for transactional, denormalized for analytical
